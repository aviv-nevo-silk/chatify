# Chatify

An Outlook add-in that re-renders an email thread as a WhatsApp-style group chat. Forwarded chains expand into separate bubbles, signatures and meeting boilerplate are stripped, and threads with mixed Hebrew / English render with per-bubble RTL.

> Reading a 17-message forwarded thread is painful. Reading the same conversation as a chat is not. That's the entire pitch.

## Install in Outlook

### Per-user sideload

1. Download the manifest:
   ```
   curl -O https://aviv-nevo-silk.github.io/chatify/manifest.xml
   ```
2. In Outlook on the Web → **My add-ins** → **Add a custom add-in** → **Add from File** → upload `manifest.xml`.

   Note: "Add from URL" is disabled on free outlook.com personal accounts. Use Add from File.

3. Open any email → "Chatify" appears in the message-read ribbon.

### Org-wide rollout

An M365 admin uploads `manifest.xml` via **Microsoft 365 admin center → Integrated apps → Upload custom apps** and assigns it to specific users or the whole tenant. The manifest declares only `ReadItem` (read-only access to the open message) — no compose, send, or delete permissions.

## What you get

- **Forwarded-chain expansion** — one Graph message containing a deeply-nested forward becomes N separate chat bubbles, sorted chronologically. The parser splits on `<b>From:</b>` markers and handles both `<hr/>` and CSS-border separator styles.
- **WhatsApp-style group header** — avatar with subject initials (deterministic gradient), thread title, and participant list.
- **Inline "added to the conversation" events** — when a message brings in new recipients, an inline chip explains it ("↗ Tal Boitel added you to the conversation"). Same metaphor as WhatsApp/Slack add-to-group.
- **Sender color coding** — deterministic hash of sender email into a ~14-color palette; consistent across reloads.
- **Per-bubble RTL detection** — first-strong-character heuristic on each message body, so a single thread with mixed Hebrew and English bubbles renders each in its natural direction.
- **Signature, meeting-boilerplate, and disclaimer stripping** — heuristic walk-back over trailing block elements removes name/title/contact lines, table-wrapped Outlook signatures, "Get Outlook for iOS" mobile sigs, Microsoft Teams meeting blocks, calendar invite footers, separator lines, social-icon `<img>` tags, and legal confidentiality notices.
- **Inline image rendering** — `<img>` tags from `https://`, `cid:`, and `data:` sources pass the sanitizer; `http:` and `javascript:` are blocked. Most Outlook-hosted images render directly.
- **Full-screen viewer** — for narrow task panes, an "↗ Open full screen" link opens a separate browser tab that mirrors the chat at full viewport width. Sync uses URL-hash encoding to bypass Chrome's Storage Partitioning between the Outlook iframe and the standalone tab.

## Local development

```bash
git clone https://github.com/aviv-nevo-silk/chatify.git
cd chatify
docker compose up -d --build
```

The Vite dev server runs at `https://localhost:3001`. On first visit, accept the self-signed cert in your browser (`Advanced → Proceed to localhost`). The cert is auto-generated by `start.sh` inside the container — `cert.pem` and `key.pem` are gitignored.

Useful URLs for local dev:

| | |
|---|---|
| `https://localhost:3001/dev.html` | Fixture-driven preview (loads sample conversations from `tests/fixtures/`) |
| `https://localhost:3001/viewer.html` | Full-screen viewer (reads from `localStorage` or URL hash) |
| `https://localhost:3001/taskpane.html` | Outlook entry point (renders nothing useful when opened directly — needs Office.js context) |

To sideload the local manifest into Outlook (replacing the public one), upload `manifest.xml` from this repo via Add from File. URLs in the local manifest point at `https://localhost:3001`, so the dev container must be running and the cert accepted.

## Tests

```bash
npm test
```

52 unit tests across 4 suites, all running in jsdom:

- `signature-stripper.test.ts` — closing salutations, contact blocks, mobile sigs, table layouts, Teams meeting boilerplate, disclaimer footers, trailing image / br elements, false-positive guards (don't strip body paragraphs that mention titles or have digit runs that look like phone numbers)
- `forward-parser.test.ts` — header detection, split semantics, address consolidation across virtual messages
- `renderer.test.ts` — end-to-end on real fixtures: bubble count, sender alignment, system events, RTL detection
- `synthetic-fixtures.test.ts` — generic harness driven by an `expected` block in each fixture (committable synthetic conversations)

## Architecture

```
   email body HTML (Office.js or Microsoft Graph)
                    ↓
   forward-parser.ts  →  expandForwardedChain()
   ▸ splits at <b>From:</b> headers
   ▸ extracts From / Sent / Subject / To / Cc per virtual message
   ▸ DOM-based body extraction (avoids string-slice corner cases)
   ▸ consolidates same-name addresses across the chain
                    ↓
   renderer.ts  →  renderConversation()
   ▸ sort virtuals chronologically
   ▸ build thread header (avatar / subject / participants)
   ▸ walk messages, emit "X added Y" events when recipients change
   ▸ for each: sanitize + strip signatures + decorate mentions
                    ↓
   sanitize.ts wraps DOMPurify with a Chatify-specific allowlist
   signature-stripper.ts walks trailing block elements heuristically
                    ↓
   DOM bubbles (`.row`, `.bubble`, `.bubble__sender`, `.bubble__content`)
```

The renderer is pure DOM construction — same code path produces the same DOM in dev page, live taskpane, and the full-screen viewer. The viewer reads its conversation from `window.location.hash` (URL hash) because Chrome's Storage Partitioning isolates the Outlook iframe's `localStorage` from a separate browser tab; URL hash is direct and partition-agnostic.

## Phases

- **Phase 1 (v1.0.0, current)** — mock dev page + live Outlook taskpane + full-screen viewer. Reads only the currently-open Graph message. Forwarded chains in that message's body expand into bubbles via the parser.
- **Phase 2 (planned)** — Microsoft Graph SSO. Fetch all messages in the same `conversationId` so normal back-and-forth threads (where each reply is a separate Graph message) render correctly, not just forwarded ones.
- **Phase 3 (planned)** — Local LLM features via Ollama. Non-English forward header parsing (Hebrew `מאת:`, German `Von:`), per-bubble TL;DR captions, action-item extraction, mixed-language translation chips. Backend: Python FastAPI proxy in front of the user's local Ollama instance.

## Permissions and privacy

- The manifest declares `<Permissions>ReadItem</Permissions>` — read access to the currently-open message only.
- All processing is **client-side**. Chatify does not transmit email content to any external service; the only network calls are loading its own static assets from GitHub Pages.
- Inline images from `https://` sources do load (typically Microsoft attachment URLs that the user is already authenticated to via their Outlook session). `http:` and `javascript:` URLs are stripped.
